    • 부스트캠프 7기 멤버십 3주차 회고

    2022-09-25

    두 번째 스프린트를 마친 3주차를 되돌아본다.

    TL;DR

    1. 너무 관심사에만 몰두한 것 같아 아쉬움이 남은 프로젝트였다.

    2. 좋은 작업은 하기 어려우니 서투른 작업이라도 공유하자.

    3. 한 주의 마무리는 목요일에 하자.

    전반적인 소감

    너무 짧았던 한 주

    일주일이 정말로 빨리 지나갔다. 왜 빨리 지나갔을까 생각해보니, 중간에 휴일이 없었던 첫 한 주 였기 때문이었을까?

    그렇다기엔 지난 시간들도 빨리 지나갔다. 그냥 시간이 빨리 흐르는 것 같다.

    개발 외적인 이벤트

    부스트캠프의 시간을 돌이켜보면, 항상 캠퍼들이 학습에 몰입할 수 있도록 지원해주는 것들이 보인다. 감사한 일이다.

    그런데 부스트캠프에는 중간에 다양한 이벤트가 있다. 그 중에서는 개발과 관련이 없(어 보이)는 그런 것들도 있다. 왜 이런 이벤트가 존재하는 걸까? 불평은 아니고, 그냥 문득 궁금해졌다.

    좋았던 것

    프로젝트 작업 나누기

    사용하면서 편했던 방법은 Milestone, Feature, Task 순으로 작업을 나누는 것이었다.

    • Milestone: 어디까지 구현할 것인지 높은 수준에서 정한다. 떠오르는 그림이나 추상적인 문구로 일단 나눈다. 예를 들어 ‘글쓰기 기능을 새로 만든다’ 같은 것이 될 수 있다.

    • Feature: 그 Milestone을 구현하는 데 필요한, 좀더 구체적인 기능이 된다. 예를 들어 ‘동작하는 글쓰기 버튼’ 정도가 될 수 있겠다.

    • Task: 가장 작은 단위로, 실제 커밋을 작성하는 단위와 동일하다. 예를 들어 글쓰기 버튼을 구현하는 데 필요한 여러 작업이 여기에 포함된다.

    GitHub 기능 이용하기

    위에서 얘기한 것을 Pull Request와 커밋으로 연결짓는다. 나중에 작업을 되돌아볼때 트래킹이 더 편해진다.

    • Task를 처리하기 전에 항상 이슈에 등록한다. 처리한 후에는 그 이슈를 참조하여 커밋을 작성한다.

    • 처리한 Task가 모여 하나의 Feature를 완성할 때마다, Pull Request로 등록한다. 나중에 Pull Request 목록을 보면, 완성된 Feature들의 역사를 볼 수 있다.

    • 이러한 Pull Request를 모아 하나의 Milestone을 달성한다.

    하나의 이슈, 하나의 커밋

    내 생각에 코드를 작성할 때 피해야 할 일은, 지금 무엇을 하는지 정확히 모르고 하는 것이다.

    이슈부터 먼저 작성하면, 내가 지금 무엇을 하는지 알고 코드 작성에 임할 수 있는 장점이 있었다.

    TDD와 닮은 면이 있는 것 같다. 하고 싶은 것을 먼저 선언하고, 그 다음 작업에 임하기. 그러다 보면 정말로 구현하고 싶었던 게 무엇인지 나중에 알게 되는 경우도 있다.

    Share Poor Work with People

    Steal Like an Artist 라는 책에 이런 구절이 있다.

    1. Do good work

    2. Share it with people

    좋은 결과를 내는 것은 정말 어려운 일이다. 그런데 부스트캠프는 그런 것을 하러 온 자리가 아니다.

    좋은 작업물 전에 어쨌든 서투른 작업물을 여러번 내놓아야 한다. 그러면 부스트캠프는 서투른 작업을 하는 자리이다.

    부스트캠프에서 무엇을 해야 하는 지 알았으니, 슬랙에 블로그 글을 공유해보기로 했다. 두 번 정도 한 것 같은데, 올해 말까지는 계속 해보는 것이 목표이다.

    아쉬웠던 것

    부스트캠프란

    피어 세션에서 오고갔던 얘기 중에 이런 말이 있었다. “많은 음식들이 차려진 상에서 먹고싶은 부분만 잘 소화해도 충분한 것이 아닐까?”

    난 이번 프로젝트에서 차려지지 않은 다른 음식을 구해서 먹었던 것 같다.

    요구사항을 보면 무엇을 공부하기를 바라는지 의도가 보이는 것들이 있었다. 2주를 충분한 시간이라고 생각해서 하고 싶었던 것을 먼저 했었다. 그러다가 정작 놓친 중요한 개념들이 적지 않았던 것 같다. 객체 지향 프로그래밍이라던가, 쿠키가 세션 같은 것들. 아쉬운 일이다.

    마무리도 시간이 필요하다

    계획 상으로는 마지막 날까지 개발 일정을 넣었고, 실제로도 그렇게 했다.

    하지만 마지막 날에 사실상 개발 시간은 없다. 그리고 개발 하기에 적절한 일정도 아닌 것 같다.

    아마 한주의 마무리는 금요일이 아니라 목요일에 준비해야 할 것 같다.
